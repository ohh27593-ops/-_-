# to-mobile-feed-mvp

기존 밥친구 운영 경험에서 출발해,  
선택 부담이 적은 **모바일 웹 기반 생활·교류 추천 서비스**로 확장한 MVP 프로젝트입니다.

이 프로젝트는 사용자가 이미 하려던 일상 활동(산책, 식사, 휴식, 배움 등)을  
사진·영상 중심의 피드 방식으로 더 자연스럽게 탐색할 수 있도록 설계했습니다.

---

## 1. Project Overview

기존 모임·이벤트 플랫폼은 사용자가 계속해서 선택하고 결정해야 하는 구조를 전제로 합니다.  
이 프로젝트는 그런 방식 대신, **부담 없이 머무르며 콘텐츠를 소비하다가 마음이 생길 때만 행동하게 하는 UX**를 목표로 합니다.

핵심 방향은 아래와 같습니다.

- 모바일 퍼스트 웹
- 한 화면 = 하나의 콘텐츠
- 스와이프 기반 흐름형 탐색
- 선택을 강요하지 않는 눈팅 중심 UX
- 사진·영상 기반의 생활 맥락 추천

---

## 2. Why This Project

기존의 모임 중심 서비스는 참여 자체가 목적이 되기 쉽고,  
사용자는 참여 여부, 시간, 장소, 사람을 계속 선택해야 합니다.

하지만 실제 일상은 조금 다릅니다.

사람들은 이미
- 어디를 걸을지
- 무엇을 먹을지
- 어디서 쉬거나 시간을 보낼지

이런 고민을 하고 있고,  
문제는 “무언가를 새로 선택하는 것”보다  
**이미 하려던 하루를 더 잘 즐길 수 있게 도와주는 정보**가 부족하다는 점입니다.

이 프로젝트는 그 지점에서 출발했습니다.  
SNS에는 일상 기록이 많지만, 그것을 사용자의 실제 행동에 도움이 되도록 다시 구조화한 서비스는 드뭅니다.  
그래서 이 MVP는 **일상 기록을 결정 보조형 피드로 재구성하는 웹 경험**을 실험합니다.

---

## 3. Core Philosophy

이 프로젝트는 아래 철학을 중심으로 설계했습니다.

### 선택을 요구하지 않는다
사용자는 처음부터 무언가를 결정할 필요가 없습니다.  
일단 보고, 넘기고, 머무르다가, 필요할 때만 더 자세히 봅니다.

### 머무르는 시간 자체가 의미가 있다
단순 클릭 수보다,  
어떤 분위기와 맥락의 콘텐츠에 오래 머무는지가 더 중요하다고 봤습니다.

### 행동은 항상 사용자가 먼저 결정한다
서비스는 강하게 밀어붙이지 않습니다.  
예약, 방문, 참여 같은 행동은 사용자가 먼저 마음이 생겼을 때만 이어지도록 설계했습니다.

---

## 4. MVP Features

현재 MVP에서 구현한 핵심 요소는 아래와 같습니다.

- **모바일 프레임 고정 UI**
  - PC에서도 모바일 화면처럼 보이도록 구성
- **세로 메인 피드**
  - 한 화면에 하나의 콘텐츠를 보여주는 구조
- **가로 상세 패널**
  - 필요할 때만 옆으로 열어 상세 정보 확인
- **스와이프 기반 탐색**
  - 세로 이동과 가로 이동을 분리한 인터랙션
- **Fullscreen Viewer**
  - 이미지/영상 몰입형 보기
- **Mock Data 기반 피드 렌더링**
  - 실제 API 연결 전 UX 검증용 구조
- **+n Prefetch 구조**
  - 다음 콘텐츠를 미리 불러와 탐색 흐름 유지
- **탭 / 검색 / 모달 UI**
  - 홈, 친구, 마이 등 확장 가능한 구조 포함
- **Google 로그인 확장 고려**
  - 현재는 localStorage 기반 MVP 세션 구조

## 5. UX Structure

이 프로젝트의 UX는 단순한 “예쁜 화면”보다  
**소비 흐름 자체를 어떻게 설계할 것인가**에 초점을 맞췄습니다.

### 한 화면 = 하나의 콘텐츠
사용자는 여러 카드 중에서 비교 선택하기보다,  
하나의 콘텐츠를 보고 다음으로 넘기는 흐름 안에서 탐색합니다.

### 세로 메인 / 가로 상세 분리
메인 피드는 빠르게 흐름을 타고,  
상세 정보는 가로 이동으로만 열리게 해서  
탐색 리듬이 깨지지 않도록 했습니다.

### 방향 잠금 기반 스와이프
세로 탐색과 가로 탐색이 섞여 오작동하지 않도록  
터치 방향을 분리하는 구조를 사용했습니다.

### 눈팅 중심 UX
처음부터 가입, 참여, 예약을 강요하지 않고  
“보기 → 머무르기 → 관심 생기면 상세 보기”의 흐름을 우선시했습니다.

---

## 6. Code Structure

현재 MVP는 **단일 HTML 파일 기반 프로토타입**으로 구현되어 있습니다.

주요 구조는 다음과 같습니다.

- `header`
  - 브랜드 / 검색창
- `main`
  - 세로 피드 영역
- `nav`
  - 하단 탭바
- `vtrack / vpage`
  - 메인 세로 콘텐츠 구조
- `htrack / hpanel`
  - 메인/상세 가로 전환 구조
- `modal`
  - 로그인 모달
- `fullscreen viewer`
  - 미디어 전체화면 보기

상태는 크게 아래처럼 나뉩니다.

- 메인 피드 상태
- 상세 피드 상태
- 세션 상태(localStorage)
- mock / net 전환 구조

이렇게 나눈 이유는 나중에 실제 서버, 추천 로직, SNS 연동을 붙일 때  
구조를 유지하면서 확장하기 위해서입니다.

---

## 7. Personalization Direction

이 MVP는 아직 완전한 추천 시스템을 붙이지 않았지만,  
향후 아래 요소들을 종합해 개인화할 수 있도록 방향을 잡았습니다.

- 웹사이트 내 체류 시간
- 탐색 패턴
- 현재 위치
- 자주 보는 카테고리
- SNS 업로드 게시물의 분위기와 주제
- 친구의 방문/기록/참여 이력

즉, 단순히 “사람이 많이 간 곳”이 아니라  
**특정 사용자에게 더 잘 맞는 장소와 활동의 맥락**을 추천하는 방향입니다.

---

## 8. Business Direction

이 프로젝트는 단순한 핫플 소개 서비스가 아닙니다.  
핵심은 **생활·교류 추천 플랫폼**이라는 점입니다.

주요 방향은 다음과 같습니다.

- 일상 활동 보조 중심 서비스
- 기록을 행동에 도움 되는 정보로 재구성
- 광고/할인보다 자연스러운 참여 유도
- 장소 제휴, 클래스 연결, 지역 콘텐츠를 통한 수익화 가능성 검토
- 장기적으로 앱(WebView) 및 개인화 알고리즘 고도화 고려

---

## 9. What Makes This Different

### 기존 모임 서비스와의 차이
기존 서비스가 “참여할 모임을 고르게 하는 구조”라면,  
이 프로젝트는 “원래 하려던 일상을 더 잘 보내도록 돕는 구조”를 지향합니다.

### SNS와의 차이
SNS가 기록 공유에 가깝다면,  
이 프로젝트는 그 기록을 **선택 부담이 적은 추천 피드**로 다시 정리하려고 합니다.

### 일반 핫플 서비스와의 차이
핫플을 단순히 사람이 많은 곳으로 보지 않고,  
**특정 사용자에게 사회적 교류와 일상이 자연스럽게 이어지는 공간**으로 해석합니다.
